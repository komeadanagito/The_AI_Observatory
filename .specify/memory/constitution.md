# 灵境观星台 (The AI Observatory) 宪法

## 项目愿景

构建一个 AI 驱动的玄学洞察平台，融合东西方神秘学体系，为用户提供个性化的命理分析与人格探索服务。

### 四大核心系统

| 系统 | 目录 | 核心特征 | 解读方式 |
|------|------|----------|----------|
| AI 塔罗牌 | `Tarot/` | 图像意象与直觉 | 视觉符号 + AI 联想解读 |
| AI 中式算命 | `Chinese_Astrology/` | 历法推算与逻辑推演 | 八字/紫微斗数算法 + AI 诠释 |
| 星座系统 | `Constellation/` | 西方占星学 | 星盘计算 + AI 性格分析 |
| MBTI | `MBTI/` | 人格类型理论 | 量表测评 + AI 深度解析 |

### 共享基础设施

- **认证系统**: 统一登录/注册，支持多种认证方式
- **用户管理**: 跨系统统一的用户数据与偏好存储
- **历史记录**: 所有解读结果的持久化与回溯

## 核心原则

### I. 模块独立性 (Module Independence)

每个玄学系统 MUST 作为独立模块运行，具备以下特性：

- **自包含**: 每个系统目录包含完整的业务逻辑、数据模型、API 接口
- **独立测试**: 系统可脱离其他模块单独运行测试套件
- **清晰边界**: 系统间通信仅通过统一用户服务，禁止直接依赖
- **可插拔**: 新系统可在不影响现有系统的前提下接入

**理由**: 玄学系统方法论差异显著，解耦设计确保维护性与可扩展性。

### II. AI 驱动解读 (AI-Driven Interpretation)

所有解读功能 MUST 通过 AI 模型生成，遵循以下规范：

- **Prompt 模板化**: 每类解读 MUST 有对应的 Prompt 模板文件
- **上下文注入**: 用户历史、偏好、当前状态 MUST 纳入 Prompt 上下文
- **结果结构化**: AI 输出 MUST 解析为结构化数据（Pydantic 模型）
- **可追溯**: 原始 Prompt 与 AI 响应 MUST 记录用于审计与优化

**理由**: AI 是平台核心价值，规范化确保解读质量可控、可迭代。

### III. 统一用户数据 (Unified User Data)

跨系统用户数据 MUST 遵循单一事实来源原则：

- **单一用户模型**: 所有系统共享统一的 User 实体定义
- **扩展字段**: 系统特有数据通过关联表扩展，禁止修改核心 User 模型
- **数据隔离**: 各系统解读记录独立存储，通过 user_id 关联
- **隐私边界**: 敏感数据（生日、出生时间）加密存储，按需解密

**理由**: 统一用户体验与数据一致性是多系统平台的基础。

### IV. 配置解耦 (Configuration Decoupling)

所有配置数值 MUST 外部化管理：

- **环境变量**: API Key、模型端点、数据库连接串等 MUST 写入 `.env`
- **禁止硬编码**: 代码中禁止出现配置字面量（端口、阈值、密钥等）
- **动态路径**: 文件访问 MUST 使用 `pathlib` 动态计算，禁止绝对路径
- **环境分离**: 开发/测试/生产环境通过 `.env.{environment}` 区分

**理由**: 配置解耦是可部署性与安全性的基础。

### V. 契约优先 (Contract-First)

功能实现 MUST 遵循接口先行原则：

- **Protocol 定义**: 模块公开接口 MUST 先定义 Protocol/Interface
- **数据模型**: 输入/输出 MUST 使用 Pydantic 模型定义，含字段语义说明
- **边界条件**: 异常情况、边界触发条件 MUST 在接口文档中明确
- **显式依赖**: 模块间依赖仅通过依赖注入，禁止全局变量传递

**理由**: 契约驱动降低耦合，确保模块可独立演进。

### VI. 中文规范 (Chinese Documentation Standard)

项目文档与代码注释 MUST 使用中文：

- **代码注释**: 函数、类、模块注释一律使用中文
- **Commit 消息**: Git 提交消息使用中文描述
- **技术文档**: spec、plan、tasks 等规范文档使用中文
- **用户界面**: 前端文案默认中文，国际化按需扩展

**理由**: 目标用户群体为中文用户，中文文档降低沟通成本。

### VII. 测试驱动 (Test-Driven Development)

功能开发 MUST 遵循 TDD 流程：

- **测试先行**: 编写实现代码前 MUST 先编写失败的测试用例
- **Red-Green-Refactor**: 严格遵循 红(失败)-绿(通过)-重构 循环
- **覆盖要求**: 核心业务逻辑测试覆盖率 MUST ≥ 80%
- **集成测试**: 系统间交互、API 契约 MUST 有对应集成测试

**理由**: TDD 确保代码质量，降低回归风险。

### VIII. 终端操作审批 (Terminal Operation Approval)

涉及终端/命令行操作时 MUST 遵循人工审批流程：

- **禁止自动执行**: AI 助手 MUST NOT 自动执行终端命令
- **指令输出**: 所有终端操作 MUST 以可复制的命令格式输出给用户
- **等待确认**: AI 助手 MUST 等待用户执行命令并反馈结果后再继续
- **错误处理**: 用户反馈错误时，AI 助手 MUST 分析错误并提供修正方案

**适用范围**:
- 包管理操作 (`uv add`, `npm install` 等)
- Git 操作 (`git commit`, `git push` 等)
- 数据库迁移 (`alembic upgrade` 等)
- 服务启动/停止 (`uvicorn`, `docker` 等)
- 任何可能影响系统状态的命令

**例外情况**:
- 只读查询命令（如 `ls`, `cat`, `git status`）可由 AI 助手执行
- 用户明确授权自动执行时

**理由**: 终端操作可能产生不可逆影响，人工审批确保操作安全可控。

## 开发流程 (Development Workflow)

### 功能开发流程

1. **规范定义** (`/speckit.specify`): 编写功能规范，明确用户故事与验收标准
2. **实现计划** (`/speckit.plan`): 技术研究、数据模型、API 契约设计
3. **任务拆解** (`/speckit.tasks`): 生成可执行任务清单
4. **TDD 实现**: 测试先行，红绿重构
5. **代码评审**: PR 必须通过宪法合规检查

### 代码评审检查点

- [ ] 是否符合模块独立性原则
- [ ] AI 解读是否有 Prompt 模板
- [ ] 配置是否全部外部化
- [ ] 是否定义了数据契约（Pydantic 模型）
- [ ] 测试覆盖率是否达标
- [ ] 注释是否使用中文
- [ ] 终端操作是否经过用户审批

## 治理规则 (Governance)

### 宪法效力

- 本宪法 MUST 优先于所有其他开发实践
- 任何代码评审 MUST 验证宪法合规性
- 违反宪法的代码 MUST 在合并前修正

### 修订流程

1. 提出修正案，说明变更理由与影响范围
2. 更新宪法文档，递增版本号
3. 同步更新所有依赖模板（plan、spec、tasks）
4. 发布变更日志，通知所有贡献者

### 版本策略

- **MAJOR**: 原则删除/重定义（破坏性变更）
- **MINOR**: 新增原则/章节，扩展指导
- **PATCH**: 措辞澄清、笔误修正

### 例外处理

复杂性例外 MUST 在 `plan.md` 的 Complexity Tracking 表格中记录：

| 违规项 | 必要性说明 | 拒绝更简单方案的原因 |
|--------|------------|----------------------|
| 示例：第4个子模块 | 当前需求 | 为何3个模块不足 |
